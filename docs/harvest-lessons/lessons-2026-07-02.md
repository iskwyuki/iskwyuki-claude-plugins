## 教訓一覧

### L1. API エラーレスポンスに内部例外の文字列を含めない

`detail=f"...: {exc}"` のように例外オブジェクトをそのまま応答へ埋める習慣は API 全面に広がりやすく、後から一掃修正になる。内部情報（スタック・接続文字列・内部パス）の漏洩経路にもなる。詳細はログへ、クライアントには定型メッセージのみ返す。

- 検出可能: `grep -nE 'detail=f".*\{(e|exc|err)' ` 系の機械チェックが効く → **hooks / CI 昇格候補**

### L2. 固定パスのルートはパスパラメータのルートより先に登録する

`GET /items/{id}` を先に登録すると `GET /items/special-action` のような固定パスがパラメータとして飲み込まれ、ID のパースエラーになる。登録順マッチのフレームワークでは「固定パス → パラメータパス」の順を守る。

### L3. 並行収集の内側で共有 DB セッションを使わない・最新値の引き直しは並行部の後で一括に

`gather` 等の並行タスク内から同一の非スレッドセーフなセッションに触れると競合する。また項目ごとに DB を引き直すと N+1 になる。並行部の内側はスナップショット値で完結させ、表示用の最新値は並行完了後にヒット ID 群への一括クエリで引き直す。引き直し失敗時はスナップショットで継続する（可用性優先のフォールバック）。

### L4. 複製・派生レコードへの編集伝播は「全フィールド」を原則にする

正本レコードの編集を派生レコード（分割保存された参照行・非正規化コピー等）へ一部のフィールドだけ伝播すると、残りのフィールドが旧値のまま表示され続ける。伝播対象はフィールド列挙を一箇所に定義し、編集経路（個別・一括・インポート）すべてから同じ伝播処理を通す。回帰テストを添える。

### L5. バックグラウンド掃除ジョブには 3 つのガードを備える

孤立データの自動削除は (a) 進行中ステータスの除外、(b) 作成直後の猶予期間（外部反映のタイムラグ中を孤立と誤判定しない）、(c) 短時間の重複実行防止 — の 3 点が揃っていないと、処理中のデータを消す事故につながる。

### L6. イベントループ内で同期 SDK・ブロッキング I/O を直接呼ばない

非同期サーバー内で同期クライアントの一覧取得＋逐次 retrieve のような処理を直接呼ぶとイベントループ全体が塞がる。ワーカースレッド（`to_thread` 等）へ逃がす。

### L7. 「最高値・累計」系の表示を直近イベントの値で上書きしない

ベストスコア等の集計表示に「今回の値」を代入すると、成績が下がったときに表示まで降格する。保存済みの集計値を参照し、直近値は集計値が無い場合のフォールバックに留める（`stored ?? current`）。

### L8. 集計の分子と分母は同一のスコープ条件で絞る

進捗率・正答率のような比率算出で、分母だけ対象範囲で絞り分子を絞り忘れると、要素が複数グループに跨る構成（多対多）で対象外メンバーの実績が分子に混入する。分子・分母・関連集計（時間・回数）に同じスコープ述語を適用し、跨り構成の回帰テストを書く。

### L9. E2E テストは「非決定な選択」と「DOM 構造依存」を避ける

順不定データへの `.first()` 依存、親要素への構造参照（`locator("..")`）は flaky の温床。ロールベースのセマンティックなロケータ＋条件フィルタで対象を一意にする。あわせてレビューで繰り返し指摘される観点: route mock の明示、hydration 完了待ち、timeout の明示、null ガード、クリック座標の動的計算。

### L10. 「統一」修正が繰り返されるなら、散在そのものを直す

文言の揺れ・同種 UI コンポーネントの混在・エンコード処理の漏れ、といった「統一」系修正が多発するのは、同じ関心事が複数箇所に散っているのが根因。共通コンポーネント／ユーティリティへ集約し、用語は用語集として rules 化する。同族のレビュー頻出観点: 型キャストでコンパイラを黙らせず型を絞る、データ取得は共通クライアント経由に統一する。
