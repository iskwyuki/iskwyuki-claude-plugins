# pr-review-loop 実走計測と自動マージ昇格判断

Plans.md **3.1** の記録。実 PR で pr-review-loop（`/code-review` マルチエージェント + 反証検証パス → 修正 → 再レビュー）を回し、指摘数・棄却数・誤修正数を測って、**条件付き自動マージ**（CI green ＋ 検証済み Critical ゼロ ＋ 収束）への昇格可否を判断する。

計測: 2026-07-04 / モデル: レビュー役・検証役ともに Opus 4.8（brain 固定。[operations.md](operations.md) の role-scoped 方針どおり）。被レビュー成果物（実装）は Fable 5 / Opus 4.8 が混在。

## 指標の定義

| 指標 | 定義 |
|---|---|
| 指摘数 | レビューエージェントが上げた指摘の総数（重複統合後・Critical/Warning のみ。Info/Good は除外） |
| 棄却数 | 反証検証パスで refuted となり統合レポートから除外した数（＝ recall 過剰を precision 側で回収した数） |
| 修正数 | 実際に修正した指摘数 |
| 見送り数 | confirmed だが medium/構造的/ローカル修正不能で見送った数 |
| **誤修正数** | 修正したが後に「不要・誤り」と判明し revert/再修正した数（＝ **false-fix**。自動マージ昇格の最重要指標） |

## 昇格判断（2026-07-04）

**判断: 条件付き自動マージへは昇格しない。マージは手動を維持する。**

誤修正率 0%（full 3 件・17 修正）は良好で DoD の件数要件（3 件）も満たしたが、以下の理由で自動マージの信頼性根拠としてはまだ不足:

1. **サンプルが docs/config に偏り**: full 実走 3 件はいずれも docs/config 変更で、false-fix リスクが最も高いアプリケーションロジックの実績がゼロ。この偏りが解消されない限り 0% は「低リスク変更での 0%」に留まる。
2. **レビューの可用性が自動マージの前提だが、実際に落ちた**: session 上限でマルチエージェントが起動不可になる事象が本計測中に発生（degraded #8/#9）。「CI green ＋ 検証済み Critical ゼロ」を自動で判定するには、レビュー基盤が安定して回ることが前提になる。
3. **実走 #2 で本番実害の Critical を 2 件検出**: 自動マージ（＝レビューをゲートにしない運用）だったら壊れたビルドが出ていた。レビューの価値が高い＝レビューをスキップする自動マージのリスクが高い、という両面。
4. **実走 #3 で書き手自身の集計誤りを検出**: レビューは自分（Lead）の doc の算術矛盾すら捕捉した。人手の見落としが常在する以上、その検出役を無くす自動マージは時期尚早。

### 昇格の再評価条件

docs/config での 3 件は満たしたので、次のすべて（＝残る不足）を満たしたら再評価する:

- **アプリケーションコード**（docs/config でない）の PR で full マルチエージェント実走を **3 件以上**（現状ゼロ。最大の不足）
- それらで**誤修正率が継続して 0% 近傍**（1 件でも false-fix が出たら閾値・修正条件を見直す）
- レビュー基盤（サブエージェント or 代替）が**安定して起動できる**運用の確立
- 昇格しても、初期は「自動マージ提案 → 人間が最終承認」の半自動から始め、full 自動化は段階的に

それまでは [operations.md](operations.md) のマージ方針どおり、pr-review-loop は収束後「マージ可能」と報告し、セルフマージ自己点検を経て**手動マージ**する。
