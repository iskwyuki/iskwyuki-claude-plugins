# 品質基準レビューセット（quality-baseline）

モデル非依存のレビュー資産（/code-review・rules・agents）の検出力を測る「物差し」。実在リポジトリの既知バグから構築した基準セットに対してレビューを実行し、検出率を比較することで、モデル交代・資産変更による品質の劣化・改善を定量評価する。

初回基準: **Fable 5 ＋ 資産一式**（2026-06 計測）。以後、モデルや資産を変えて同一セットを流し、検出率を比較する。

## ケースの構成

ケースは公開・非公開の 2 系列で、**方法論・判定基準・集計フォーマットは共通**。

| 系列 | 出所 | 保存場所 |
|---|---|---|
| PF-* | 公開リポジトリ（iskwyuki/portfolio 等） | 本ディレクトリ `cases/`（raw 情報込み） |
| CO-* | 非公開リポジトリ | ローカル専用 manifest（`~/dev/quality-baseline-private/`、git 管理外）。本ディレクトリにはカテゴリ・重大度・難易度と集計数字のみ掲載 |

非公開ケースの固有情報（リポジトリ名・SHA・コード・根本原因の原文）は、GitHub 上のリポジトリには一切置かない。

## ケースの定義（重要）

各ケースは 1 つの既知バグで、次の 3 点で構成する。

1. **導入コミット（レビュー対象）**: バグを混入させたコミット。**レビューはこのコミットの diff に対して実行する**（修正コミットの親ではない。バグは修正直前ではなく導入時に捕まえるべきものであるため）
2. **修正コミット**: バグが実際に直されたコミット（根本原因の裏付け）
3. **正解**: ファイル・関数（行範囲）・根本原因 1 行・重大度

## 選定基準

- レビューで検出可能な**実装バグ**に限定（仕様変更・仕様誤解・外部環境起因は除外）
- **Linter / 型チェッカー / フォーマッタが機械検出できるものは除外**（例: ESLint ルールで捕まる set-state-in-effect、tsc で落ちる型エラー）。このセットは「人間/LLM レビューでしか捕まえられないバグ」専用
- upstream ライブラリ起因の回避策（モンキーパッチ等）は除外
- カテゴリ 7 種（下記）×難易度（易/中/難）を分散させ、モデル間の差が出る構成にする

## カテゴリと重大度

カテゴリ: `ロジック` / `型・null` / `状態管理` / `hydration` / `非同期・競合` / `セキュリティ・認可` / `設定・ビルド`

重大度は実務リポジトリのレビュー運用に合わせて `Critical / High / Medium / Low`。観点タクソノミー（アーキテクチャ・コード品質・セキュリティ・パフォーマンス・テスト・設定・潜在的バグの 7 観点）も同運用から採用しており、検出結果の観点別分析に使う。

## 判定基準（2026-06-12 確定）

レビュー出力を正解リストと突き合わせ、ケースごとに次の 4 区分で記録する。

| 区分 | 定義 |
|---|---|
| 検出 | 同一ファイルかつ同一根本原因を指摘（行番号・表現のズレは許容） |
| 部分検出 | 場所は当てたが原因が異なる、または原因に触れたが場所が不正確 |
| 見逃し | 該当指摘なし |
| 追加指摘 | 正解リスト外の指摘（後で真偽判定し、誤検知率の分子にする） |

## 再実行手順

1. 対象ケース一覧を読み込む: 公開分は `cases/`、非公開分はローカル manifest（`~/dev/quality-baseline-private/cases.md`）
2. ケースごとに**導入コミットの diff** を取得する
   - 公開分: `git -C <repo> show <導入SHA>`
   - 非公開分: manifest 記載のワンショット認証手順で `gh api .../commits/<導入SHA>` から取得（clone しない）
3. 測りたいレビュー資産（例: `/code-review full`）に diff を入力し、レビュー結果を保存する
4. 判定基準 4 区分で採点し、`results/` に「モデル×資産バージョン」単位で記録する（フォーマットは `results/template.md`）
5. 過去の結果と検出率・誤検知率を比較する
