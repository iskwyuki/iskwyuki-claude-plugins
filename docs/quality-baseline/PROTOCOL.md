# baseline-protocol v1 — 品質基準セット再走プロトコル

品質基準セット（[README.md](README.md)）を、**モデルや実行者が変わっても同一条件**で再走するための固定手順。Fable 5 baseline（[`results/2026-06-13-fable5-assets-0.5.0.md`](results/2026-06-13-fable5-assets-0.5.0.md)、検出 8/13＝61.5%）はこの v1 で計測した。**以後の比較計測は必ず本プロトコルで実施する**（プロトコルを変えると過去計測と比較できなくなる）。

- README.md = 方法論・ケース定義・判定基準
- 本ファイル = 再走の実行手順（ブラインド条件・agent プロンプト・採点・チェックリスト）

---

## 0. 不変条件（v1 の同一性を保つ規約）

| # | 規約 | 理由 |
|---|------|------|
| 0-1 | **ブラインド**: レビュー実行 subagent には「対象リポジトリと導入コミット SHA」のみを渡す。正解・既知バグの有無・選定理由・カテゴリ/難易度は開示しない | 後知恵・当てにいく挙動の排除 |
| 0-2 | **参照禁止**: 導入コミットより後の履歴・PR・Issue・修正コミットを参照させない | 同上（答えが未来にある） |
| 0-3 | **入力は導入コミットの diff**（修正コミットの親ではない） | バグは導入時に捕まえるべきもの |
| 0-4 | **2 agent 縮約**（下記 §1）を固定。除外 agent と理由も固定 | Full 7 並列は Issue/rules/型/簡素化が計測ノイズになるため |
| 0-5 | **検証パス（反証）は採点側が実施**。SKILL の Step5 を subagent 側では回さない | recall（実行 agent）と precision（採点者）を分離 |

## 1. 2 agent 縮約の定義（固定）

`/code-review full` の 7 agent を、ケースごとに独立起動する 2 agent へ縮約する。観点文は [`assets/skills/code-review/SKILL.md`](../../assets/skills/code-review/SKILL.md) から**逐語転記**（§2/§3 にコピペ可の形で掲載）。

| agent | 統合する Full 観点 |
|---|---|
| **X** | code-reviewer ＋ design-checker |
| **Y** | silent-failure-hunter |

**除外 agent とその理由（固定）**:

| 除外 | 理由 |
|---|---|
| requirements-checker | ブラインド条件で Issue 文脈を渡さないため機能しない |
| rules-checker | `.claude/rules/` は計測対象外（資産のレビュー観点そのものを測る） |
| type-analyzer | 型チェッカーが機械検出する領域はケース選定で除外済み |
| simplifier | 簡素化提案はバグ検出と無関係 |

## 2. X agent プロンプト（逐語・コピペ可）

```
あなたはコードレビュー担当です。以下の diff（あるコミットの変更内容）をレビューしてください。

## ブラインド条件（厳守）
- レビュー対象は与えられた diff のみ。これは対象リポジトリの、ある1コミットです
- このコミットより後の履歴・PR・Issue・修正内容は参照しないこと
- 既存ファイル・型定義・規約（.claude/rules/）を参照する場合も、この導入コミット時点のツリーに限る
- 既知バグの有無・正解・観点の当たりは与えられません。あなたの役割は diff から問題を網羅的に洗い出すことです

## 観点（code-reviewer）
- ロジックのバグや誤り
- セキュリティ脆弱性（XSS, インジェクション, 認証漏れ）
- パフォーマンス問題（N+1, 不要な再レンダリング, メモリリーク）
- エラーハンドリングの欠如
- 機密情報のハードコード

## 観点（design-checker）
- プロジェクトのアーキテクチャ・設計パターンに従っているか
- 既存コードと一貫性のある命名・構造になっているか
- 責務分離は適切か（ファイル・関数の粒度）
- .claude/rules/ のプロジェクト規約に違反していないか
- DRY: 既存のユーティリティや関数を再利用しているか

## 報告方針（最重要）
あなたの役割は網羅（カバレッジ）であり、フィルタリングではない。確信が持てない指摘・軽微に見える指摘も省略せず報告すること。誤指摘は下流の検証パスが棄却するが、あなたが黙って落とした指摘は誰にも回収できない。各指摘に confidence（high / medium / low）を付与すること。

## 出力フォーマット
### Critical（必ず修正すべき）
- `file:line` — 指摘内容（confidence: high|medium|low）
  → 修正案: ...
### Warning（推奨）
- `file:line` — 指摘内容（confidence: high|medium|low）
  → 修正案: ...
### Info（参考）
- `file:line` — 指摘内容
### Good（良い点）
- 良い設計判断・実装のピックアップ
指摘がない場合は「指摘事項なし」とだけ報告してください。
```

## 4. 採点（実行者側・非ブラインド）

X/Y の出力を受け取ったら、採点者（Lead）が以下を行う。

1. **反証検証**: 各 agent の Critical / Warning を反証プロンプトで裏取りする（SKILL.md Step5 と同一）。
   ```
   以下のレビュー指摘を「反証」せよ。擁護ではなく反証を試みること。
   実際のコード・型定義・呼び出し元を Read / Grep で確認し、指摘が現行コードに
   当てはまるかを判定し、verdict を返せ:
   - confirmed: 指摘は正しい（根拠の行番号を添える）
   - refuted: 誤指摘（反証の根拠を必ず添える）
   - uncertain: 判断不能（何が分かれば判断できるかを添える）
   ```
2. **正解突合 → 4 区分判定**（README.md の判定基準）:
   - 検出 / 部分検出 / 見逃し / 追加指摘
   - 判定は「同一ファイルかつ同一根本原因で検出成功（行・表現ズレ許容）」
3. **追加指摘の真偽判定** → 誤検知率の分子にする（run1 では未実施。可能なら Critical 帯から実施）
4. 集計を [`results/template.md`](results/template.md) のフォーマットで記録

## 5. 記録と比較基準値

出力先: `results/<YYYY-MM-DD>-<model>-assets-<plugin version>.md`（既存 Fable baseline と同じ `docs/quality-baseline/results/`）。

**比較の基準値（Fable 5・2026-06-13・資産 0.5.0）**:

| 指標 | 値 |
|---|---|
| 検出率（検出 / 全 13 ケース） | **8/13（61.5%）** |
| 部分検出込み | 10/13（76.9%） |
| 難易度別（検出 / 部分込み）: 易 / 中 / 難 | 1/1・1/1 / 6/8・7/8 / 1/4・2/4 |
| カテゴリ別ワースト | 設定・ビルド 0/3 |
| Critical 較正 | 2/2 一致 |

**Fable の弱点3点（資産改善候補・比較時の着目点）**:
1. diff 外の実行環境・適用条件系が全滅圏（設定・ビルド 0/3）
2. アクセス制御既定値のフェイルオープンを追認（fail-close 原則の未適用）
3. 検出しても重大度を低く付ける傾向（中位帯）

別モデルがこの3点で改善／悪化するかが弁別ポイント。

**公開のみ継続の基準値（会社凍結後・2026-07-05〜／Plans 4.3）**:

会社ケース凍結後の継続 recall は公開ケース PF-1〜8 のみで測る。初回公開 baseline は [`results/2026-07-05-opus4.8-public-assets-0.11.0.md`](results/2026-07-05-opus4.8-public-assets-0.11.0.md)（Opus 4.8・資産 0.11.0）:

| 指標 | 値 |
|---|---|
| 検出率（検出 / 全 8 公開ケース） | **3/8（37.5%）**（PF-4・PF-5・PF-8） |
| 部分検出込み | **5/8（62.5%）**（＋PF-2・PF-3） |
| degradation bracket（Haiku 4.5・Opus 完全検出 3 ケース） | 3/3 保持だが各 1/2 agent の辛勝（他方は偽陰性） |

> 会社混在 13 ケースの Fable 61.5%/Opus 53.8% とは**母数が異なり直接比較不可**。以後の公開再走はこの公開 8 ケース値を基準に比較する。公開のみ月次再走は `scripts/quality-monthly.sh`（3.3）に配線済み（版・モデル変化時に自動で要否判定→再走 draft を NAS レポートに保存、反映は手動）。

## 6. 実行チェックリスト（4.1 = 会社ケース CO-* を含む単発再走）

> **4.1 は Mac 専用**。会社 SSO トークン・private manifest（`~/dev/quality-baseline-private/`）は Mac にしか無い（NAS には置かない方針）。会社アクセスは資産作成フェーズ限りの特別許可で、この再走は初期 baseline 比較のための**単発例外**（恒久の定期取得禁止＝3.3 は維持）。

- [ ] 実行ホストが Mac（`~/dev/quality-baseline-private/` が存在すること）
- [ ] 会社アクセスは**ワンショット**（`GH_TOKEN=$(gh auth token -u <会社アカウント>) gh api ...` を1回だけ）。`gh auth switch` は使わない
- [ ] **read-only・clone なし・痕跡残さない**（GitHub API のみ、write 系一切なし）
- [ ] CO-1..CO-12 の導入コミット SHA を manifest から取得し、`gh api repos/<org>/<repo>/commits/<SHA>` 等で diff 取得
- [ ] PF-1（公開）も同一プロトコルで再走
- [ ] 各ケースを X/Y の blind subagent で起動 → 採点者が反証検証 → 4 区分判定
- [ ] モデル: 最低 Opus 4.8（強）。可能なら Sonnet / Haiku（弱）で degradation を bracket
- [ ] Fable 61.5% と比較（全体 / 部分込み / 難易度別 / カテゴリ別）
- [ ] `results/2026-07-04-<model>-assets-0.10.1.md` に記録
- [ ] **サニタイズ検証**: 公開 results を grep し「固有名 0 件・SHA 0 件・コード 0 件」を確認して results に明記
- [ ] `git add` は明示パスのみ（`-A` / `.` 禁止）
- [ ] **本日中に完了**（会社アクセス時限）

**サニタイズ grep の例**（results 追加前にコミットゲートとして走らせる）:

```bash
# コミット対象の公開 results に SHA 断片・固有識別子が混入していないか
git diff --cached -- docs/quality-baseline/results/ \
  | grep -nEi '[0-9a-f]{7,40}|<会社 org 名>|<会社 repo 名>' \
  && echo "NG: サニタイズ違反あり" || echo "OK: 固有情報なし"
```

> `<会社アカウント>` `<org>` `<repo>` `<会社 org 名>` 等は公開ファイルに実名を書かないためのプレースホルダ。実名は private manifest とローカルメモリのみが持つ。
