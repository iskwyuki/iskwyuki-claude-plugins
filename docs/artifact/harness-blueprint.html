配布ハーネスの設計図

個人リポジトリ群に
モデル非依存の開発品質を配布する基盤

1 つの配信元リポジトリが、skill・agent・hook・品質ゲートを marketplace 経由で各プロジェクトへ展開する。目的は「賢いモデルへの依存」を「焼き込まれた資産と検証構造」に置き換えること。

賭けの核心Fable が使えなくなっても Fable 級のアウトプット品質を保つ。そのために、レビューの観点（rules・agents）と検証構造（全件報告＋実行での裏取り）を、モデルではなくリポジトリの資産として残す。

品質ゲートの仕組み

モデルに依存しない「機械の関所」

pre-commit ゲートは、どのモデルがコードを書いても同じように機械的にバグを止める。ここがモデル非依存品質の中核。

1 · 生成

構成マーカー（package.json / pyproject.toml / Cargo.toml）を検出し、そのリポジトリで現状 PASS するチェックだけを段に入れる（未設定ツールは入れない）。

2 · 配線

prepare で git config core.hooksPath .githooks || true。|| true は git/.git 不在の Docker・CI で install を壊さないためのガード。

3 · 実行権限

git update-index --chmod=+x で tree mode を 100755 に確定。非実行だと git がフックを無音でスキップする。

4 · 発火

全コミットで tsc --noEmit や vitest を実行。Claude 経由でも手動でも git ネイティブ層で必ず通る。

今回のレビューが実際に捕まえた実害バグ①exec-bit の tree 非実行記録でゲートが無音スキップ ②prepare が Docker で pnpm install を破壊 ③集計 docs 自身の算術矛盾。いずれも「モデルの賢さ」ではなく実行での裏取り（git ls-tree を叩く・非 git dir で git config を試す）で検出された ── これが狙っていたモデル非依存の検出構造。
